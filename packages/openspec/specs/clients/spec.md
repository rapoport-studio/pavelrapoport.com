# clients

> People who bring money. Pavel brings them product.

## Purpose

**For whom:** Pavel (manages relationships), clients (eventually —
portal with project status).

**Why it exists:** A client is not a project. One client can have
multiple projects. Need to track who they are, how they came in,
what they need, what stage the relationship is at.

## Requirements

### Requirement: Client Profiles

The system SHALL store client profiles with contact information
and source of acquisition.

#### Scenario: New client created
- **WHEN** a new client is registered (manually or via AI bot)
- **THEN** the system stores name, company, contact, source
- **AND** sets initial status to "lead"

### Requirement: Relationship Lifecycle

The system SHALL track client relationship status through stages.

#### Scenario: Status progression
- **GIVEN** a client with status "lead"
- **WHEN** Pavel accepts the client
- **THEN** status changes to "active"
- **AND** the full lifecycle is: lead → active → completed → returning

### Requirement: Client History

The system SHALL maintain complete history per client —
all projects, conversations, and invoices.

#### Scenario: Viewing client history
- **WHEN** Pavel opens a client profile
- **THEN** he sees all linked projects, message threads, and invoices

### Requirement: Qualification

The system SHALL support fit scoring for incoming leads.

#### Scenario: Qualifying a lead
- **WHEN** a new lead comes in (via `messages` or manually)
- **THEN** the system records fit score, budget range, service type

### Requirement: Client Onboarding

The system SHALL guide a client from first visit to active project
through a defined journey.

#### Scenario: Full onboarding flow

```
Step 1: Discovery (public, no auth)
  Visitor lands on pavelrapoport.com
  → sees landing page with case studies
  → starts conversation with Muse (Canvas mode)
  → describes their idea
  → domain map builds in real-time
  → session stored as canvas_session

Step 2: Lead capture (end of Canvas session)
  Muse completes discovery (max 20 messages)
  → summarizes the domain map back to client
  → asks for contact: name, email, company (optional)
  → client provides contact → Lead entity created
  → Muse: "I'll share this with Pavel. He reviews
    within 24 hours."
  → shareable link: /s/[session-id]

Step 3: Pavel reviews (Studio, auth required)
  Pavel sees new session in /studio/inbox
  → reads transcript + views domain map
  → fit score auto-generated by Muse
  → Pavel decides: Accept or Decline

Step 4a: Accept
  → Client status: lead → active
  → Pavel creates Quote (engagement type, price, milestones)
  → System sends: NDA + Quote via email (Resend)
  → Client signs NDA (DocuSign or PDF confirm)
  → Client signs Quote → Contract created

Step 4b: Decline
  → Pavel writes decline reason
  → System sends polite decline + suggestions via email
  → Client status: lead → declined

Step 5: Project kickoff (after contract signed)
  → Project created from canvas_session domain graph
  → Organization created (or existing org linked)
  → Client receives: login link to client portal (future)
  → Pavel starts shaping in Studio

Step 6: Ongoing
  → Client sees project status via portal (future) or email updates
  → Invoices sent per milestone
  → On completion: client status → completed
  → Case study created (with client permission)
```

#### Scenario: Returning client
- **GIVEN** a client with status "completed"
- **WHEN** they start a new Canvas session
- **AND** provide the same email
- **THEN** Muse recognizes them: "Welcome back!"
- **AND** status changes to "returning"
- **AND** previous projects visible to Muse as context

#### Scenario: Client referred by someone
- **GIVEN** a new visitor mentions a referral
- **WHEN** Muse captures this in conversation
- **THEN** source is set to "referral" with referrer name
- **AND** Pavel sees referral source in inbox

## Entities

- **Client** — a person or company that pays for work
- **Lead** — an unqualified incoming contact

## Dependencies

- `projects` — client's projects
- `messages` — communication history
- `finance` — invoices and payments
